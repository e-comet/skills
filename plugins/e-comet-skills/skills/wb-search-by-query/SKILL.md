# Поисковая выдача WB

## Контракт

Скилл получает поисковую выдачу Wildberries по одной или нескольким фразам через браузер
пользователя: MCP tool `browser_job` подписывает задание, расширение e-Comet выполняет
запросы к WB, задание передаётся и результаты читаются через postMessage-RPC на вкладке
wildberries.ru (снипеты в `scripts/`, работают из изолированного JS-мира без CDP). Сервер
e-Comet не выполняет запросы к WB и не видит ответы — все данные остаются в браузере
пользователя. Одна страница выдачи — примерно 100 позиций.

## Требования

- **MCP сервер `e-comet`** — устанавливается автоматически вместе с плагином; при первом
  обращении откроется вход по email (код из письма).
- **Расширение e-Comet** — установлено в браузере, выполнен вход (введён ключ расширения).
- **Вход в Wildberries** — пользователь залогинен на wildberries.ru в этом браузере; задания
  без входа отклоняются (`wb_not_authenticated`).
- **Управляемый браузер** — агент должен уметь открывать вкладки и выполнять JavaScript на
  странице (page.evaluate или аналог). Без выполнения JS результаты прочитать нельзя.

## Как работает

1. Вызовите MCP tool `browser_job`:

   ```json
   { "job_type": "search_by_query", "queries": [["тушенка", 2], ["тушенка говяжья", 1]] }
   ```

   `queries` — пары `[фраза, страниц]`; суммарно не больше 50 страниц. Для вопроса «на какой
   позиции мой артикул» обычно достаточно 2-3 страниц (~200-300 позиций). Ответ:
   `trigger_url`, `job_ids` (один родительский `jobId`), `expires_at` (~5 минут — открывайте
   сразу, не откладывайте).

2. Сразу возьмите Chrome пользователя с расширением, а не встроенный (in-app) браузер агента:

   ```js
   agent.browsers.get("extension")
   ```

   Во встроенном браузере расширения e-Comet нет — задание там не выполнится.

3. Откройте в этом браузере любую вкладку wildberries.ru (достаточно главной
   `https://www.wildberries.ru/`) и передайте задание снипетом `scripts/agent_submit.js`,
   подставив в `__TRIGGER_URL__` значение `trigger_url` **как opaque-строку**: ровно как
   вернул tool — не перепечатывайте, не пересобирайте, не редактируйте и не переносите через
   многострочный текст (пробельные символы внутри = значение повреждено, запросите новый
   `browser_job`). Ответ `{ jobIds }` — задание принято, навигация на `trigger_url` не нужна.
   Ответ `{ error }` — это ошибка настройки, покажите текст пользователю как есть
   (см. «Ошибки»). Fallback, если postMessage-RPC недоступен: откройте `trigger_url`
   навигацией в той же вкладке.

4. Опрашивайте прогресс снипетом `scripts/agent_poll.js` (замените `__JOB_ID__` на
   `'<jobId>'`), раз в 2-3 секунды, до `progress.status === 'done'`. Страницы запрашиваются
   волнами с дросселем (анти-бан) — большие батчи могут занимать десятки минут; читайте
   готовые единицы по мере появления, не дожидаясь всего батча. Если вернулся `setupError` —
   прекратите опрос и покажите пользователю `setupError.message` как есть (см. «Ошибки»).

5. Прочитайте результаты снипетом `scripts/agent_read.js`, передавая в `__UNIT_IDS__`
   конкретные id порциями по 1-2 за вызов — полные тела большие (~100 товаров на страницу);
   `__UNIT_IDS__ = null` (все тела одним вызовом `readAll`) используйте только для 1-2
   страниц. Результаты хранятся ограниченное время (до суток) — забирайте их сразу. Юнит на
   каждую пару (фраза, страница): `<jobId>__q<queryIndex>__p<page>`, где `queryIndex` —
   индекс фразы в `queries` (с 0), `page` — номер страницы (с 1). В каждом юните
   `data.body.products` — упорядоченный список выдачи (плюс `data.body.metadata`), а также
   поля `query` и `page`.

## Как отвечать пользователю

- **Позиция артикула**: индекс товара в `products` (с 0) + `(page - 1) * 100 + 1`. Если
  артикул не найден на запрошенных страницах — так и скажите («не найден в первых N
  позициях»), не утверждайте, что его нет в выдаче вообще.
- **Топ по запросу**: верните первые N товаров с артикулом, названием и брендом; добавляйте
  цену/рейтинг, если эти поля есть в карточках `products`.
- Для нескольких фраз группируйте ответ по фразам.
- Юнит со `status: 'done'`, но `data.ok === false` / `data.status === 0` — сетевая ошибка WB:
  сообщите об этом по конкретной странице и предложите повторить.
- Указывайте, что выдача персонализирована слабо, но меняется во времени: позиции — снимок на
  момент запроса.

## Ошибки

Ошибки настройки — задание не принято расширением. Приходят как текст ошибки `submit` либо в
`progress().setupError`; покажите `message` пользователю как есть:

| `code` | Что сделать пользователю |
| --- | --- |
| `extension_not_authenticated` | Открыть попап расширения e-Comet и ввести ключ расширения |
| `wb_not_authenticated` | Войти в аккаунт на wildberries.ru и повторить запрос |
| `extension_misconfigured` | Обновить/переустановить расширение |
| `token_rejected` | Запросить новый `browser_job` и повторить |

Ошибки уровня юнита (`status: 'error'`, поле `error`): `Agent job timed out` (нет ответа за
таймаут), `Fetch URL host is not allowed`, `Extension connection not ready` / `closed`
(расширение недоступно — попросите проверить установку). Снипет-ответ
`{ error: 'e-Comet extension did not respond' }` означает, что расширение не отвечает на
этой вкладке — проверьте, что открыта вкладка wildberries.ru и расширение включено.
